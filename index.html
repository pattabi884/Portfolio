Privileged Access Management System

RBAC + ABAC · Async Audit Pipeline · Context-Aware Authorization · NestJS + Next.js

Most RBAC tutorials give you a role column on a users table. This isn't that. Every protected request goes through a 3-layer decision engine — JWT identity, role-based permissions against a Redis-cached layer, then a context evaluator checking business hours, IP whitelist, MFA status, and session age. All before anything goes back to the caller.

The cache layer is where it gets interesting. Role changes bust every affected user's cache in parallel via Promise.all() — no waiting for TTL. And the latest addition: time-limited access grants with auto-expiry. Approve someone's access for 2 minutes, a Redis-queued job fires at expiry and revokes it automatically. You can actually watch it happen in the demo — grant access, set the timer, and the permission disappears on its own. No manual revoke needed.

Audit pipeline runs entirely async via BullMQ so there's zero latency impact, and a background worker continuously scans for suspicious patterns — repeated denials, off-hours access, unusual frequencies — without touching the request path.

🔨 Currently building: a Go-based secrets sharing portal — paste a secret, get a one-time link, it self-destructs after opening or after N minutes. Wanted an excuse to write real Go and this felt like the right kind of small, sharp thing to build.

⏱️Time-limited access grants — approve access with an expiry window, a Redis-queued job auto-revokes when it hits zero. There's a 2-minute demo mode you can trigger in the live app — grant access, watch the clock, watch it disappear.
📋Async audit pipeline — BullMQ handles all audit writes in the background. Request latency is completely unaffected regardless of audit volume.
🚨Suspicious activity detection running entirely in background workers — off-hours access, repeated denials, unusual permission frequencies — never touches the REST path.

Realtime Geospatial Engine

Uber Backend Clone · Kafka · Redis GEO · PostGIS · WebSockets

The hard part of building something like Uber's backend isn't tracking location — it's tracking it at volume without the database becoming the bottleneck. This engine is designed around exactly that problem.

⚙️Geo-spatial Kafka sharding — driverId as partition key means all updates for a driver always land in the same partition. Location A → B ordering is deterministic, always.
👻The Ghost Driver fix — pairing GEOADD with SET EX (10s TTL). Drivers that stop heartbeating auto-evict from the proximity index. Kill the simulator: drivers vanish after 15s.
🔁Backpressure-aware consumers — lag = current time − event timestamp. If lag exceeds 2s, stale frames drop. The system always prioritises live data over catching up.
💾~99% fewer DB writes — updates buffer in memory and flush to PostGIS in batches of 100 (or every 5s). Same durability. A fraction of the IOPS.

Fintech · Distributed Database Architecture

Sharded Wallet System

DB Sharding · Event-Driven Transactions · Idempotency · Ledger Consistency

A high-performance fintech backend built to answer: what does it actually take to build a wallet that can't double-charge users, won't fall over under load, and scales horizontally?

🔀Horizontal DB sharding — data split across multiple PostgreSQL instances via Merchant ID modulo routing. Custom Prisma clients enforce a strict boundary between identity data (central DB) and transactional data (shard DBs). Cross-tenant bleed is impossible by design.
⚡Event-driven ingestion — API accepts requests, pushes to BullMQ, returns 202 Accepted immediately. Workers handle shard routing, ledger sync, and webhooks. Zero blocking on the hot path.
🔒Idempotency via IRN — every transaction carries a unique Internal Reference Number. Duplicates detected and deduplicated before touching the DB. No double-spending under any concurrency.
🪝Merchant webhook pipeline — secondary BullMQ queue fires external notifications after ledger sync, keeping integrations fully decoupled from core transaction flow.
